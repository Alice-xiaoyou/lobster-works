Google出了Vibe Coding白皮书——被3000人质疑的东西，现在有了官方定义

6月21日同一天：HN社区5篇热帖3107赞全在质疑Vibe Coding。Google发布官方白皮书定义"新的SDLC"。社区在质疑，大厂在定义。这不是矛盾——这是共识正在浮现。

Google白皮书5个核心结论

1. Agent = 10%模型 + 90%工具链

模型只是引擎。harness（指令、规则、工具、MCP服务器、沙箱、编排、可观测性）才是车身+道路+交规。有团队仅改harness就从Terminal Bench 2.0的30名外进入Top 5。LangChain只改系统提示词+工具就加了13.7分。

AI Agent犯错时，90%是你的指令有问题，不是模型不够聪明。

2. Context Engineering决定你的账单

静态上下文（每轮加载全部规则）→ 可靠但贵。动态上下文（按需加载技能/工具/文档）→ 便宜但需设计。把非结构化规则文件一股脑全塞进每轮prompt = 最贵的做法。

3. Verification = Vibe Coding ↔ Engineering的分界线

同一个Agent上，从纯vibe到完整engineering的任意位置都可以。决定位置的，是你有没有verification。

Tests覆盖确定性部分。 Evals覆盖非确定性：Output eval（结果对吗）+ Trajectory eval（推理过程对吗）。

"把标准设在eval上，不是demo上。" — Google白皮书
以上三个——Agent的结构、Context的管理、Verification的定位——是白皮书的"框架层"。下面两个，是关于成本和趋势。

4. Vibe Coding长期贵3-10倍

便宜在前面（prompt几句话出代码），贵在后面（token burn + 维护税 + 安全清理）。Agentic Engineering反转：前期投入多（schema/tests/context），每feature后续成本低。

5. 原型就是生产Agent

同一终端工作流→从throwaway脚本到production agent。天顶仍然是80%问题——最后20%的边界条件依然需要人。

3条可立刻用的结论

① 从今天起，用"人机分工线"代替"让AI做一切"
如果你只带走一条，就是这条。
🟢样板代码→AI全权 · 🔴架构安全→人拍板 · 🟡初稿审查→协作。最大陷阱：区域A膨胀——AI太擅长样板代码，你会不自觉让它做架构决策。

② 先建Verification，再提速
AI写的模块=没测试=没完成。要测结果（output eval）也要看推理过程（trajectory eval）。

③ 重新理解"学编程"
不再是写代码快。是写spec让AI准确理解+读懂AI代码的隐患+在3个方案里选最优+设计规则让Agent不自相矛盾。
