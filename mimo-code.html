小米开源MiMo Code：HN 557分/315评论——编码Agent的"长程决策"解法

6月10日，小米MiMo团队开源了MiMo Code——一个终端编码Agent，MIT许可证。HN上557分，315条评论。这是2026年迄今为止中国科技公司开源项目中HN得分最高的。

为什么还要关注一个编码Agent？

过去半年编码Agent已经很拥挤：Claude Code、Codex、Cursor、Devin……但MiMo Code解决的问题，是所有编码Agent都遇到但没公开说的：长程任务的决策质量衰减。

大多数编码Agent在10轮以内表现良好。但当Agent运行到50轮、100轮时——上下文窗口被填满、早期约束被稀释、模型开始做出越来越差的决策。

MiMo Code的设计宣言很明确：不是让Agent更聪明，是让Agent在变长的任务中不变得更笨。

三个核心设计：计算、记忆、进化

1. 计算层：Max Mode（并行采样+裁判筛选）

每轮决策时，MiMo Code让同一个模型生成5个候选方案（temperature=1，确保多样性），然后让低温度版本的模型当裁判，选出最好的。

直觉上"浪费"了5倍计算。但SWE-Bench Pro上的结果是10-20%的性能提升。

为什么管用？因为编码Agent的每一步错误都在复利。用5倍的单步计算换一个更可能正确的下一步，在长程任务中是稳赚的。

如果5个候选收敛到同一方案→高置信度。如果5个候选分歧很大→裁判选出最稳健的那个，比赌一个采样点靠谱。

2. 记忆层：显式存储-检索

常规做法是压缩历史成摘要。MiMo Code的判断是：摘要不断强化近处信息、弱化远处信息。你需要的是显式的"存什么、什么时候取"的机制。

它把关键信息写入持久化结构（文件系统、数据库、结构化记录），在需要时按需检索。这和Headroom的上下文压缩形成互补——Headroom做无损压缩，MiMo Code做有选择的记忆。

3. 进化层：跨会话体验蒸馏

每次完成任务后，从执行日志中提取经验——什么策略有效、什么模式重复出错——写入知识库。下一次任务自动注入。

野心是：Agent不只是在每个任务中"不犯同样的错"，而是"越来越像一个了解这个代码库的老工程师"。

实测验证（6月23日更新）

在文章首发24小时后，我们完成了本地安装和基本功能测试：

安装体验

零摩擦。npm install -g @mimo-ai/cli 仅2个包，10秒完成。首次运行自动SQLite数据库迁移，无需注册、无需API key、无需配置。MiMo Auto免费通道直接可用。

  • 版本：v0.1.2
  • 总成本：$0.00（226 sessions历史，全免费通道）
  • 平均：1.7M tokens/session，中位180K

测试1：简单文件创建

任务：「Create a simple hello world function in a file called hello.js」

结果：✅ 通过，~3秒。创建了 hello.js，含 hello() 函数 + CommonJS导出。代码风格干净。使用了 dream 子Agent规划 + build 主Agent执行。

测试2：多步骤代码修改

任务：给Express应用添加错误处理——try/catch包裹listen、添加/health端点、添加uncaughtException处理器。

结果：✅ 通过，~5秒。三个要求全部正确完成。没有引入额外改动，代码一致性保持完好。观察到的Agent切换流程：

  1. dream 子Agent — 并行采样多个方案（temperature=1），裁判筛选
  2. build 主Agent — 执行文件写入（edit + write工具）
  3. distill 子Agent — 从执行日志中提取经验写回记忆库

权限系统

安全设计比预期更严格：

  • 工作目录（/tmp）默认允许bash/edit/write
  • 外部目录（如 ~/.local/share/mimocode）默认拒绝
  • 每次首次操作弹出权限请求，可设白名单

这对"AI代码不可审查"问题提供了一个工程化答案——不是审查代码，是限制它能碰什么。

实测诚实声明：以上测试基于简单任务（<10行代码修改）。长程任务（50+轮）的Max Mode效果、复杂工程场景的表现、非MiMo模型后端的兼容性——这些仍然标注未验证。MiMo Auto免费通道锁定mimo-auto模型，无法切换后端测试。
