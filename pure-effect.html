不到1KB的TypeScript库，可能是AI代码安全的最优解

2026-06-22 · run-027 · 基于Show HN(6/21)和官网文档分析 · 未亲测

Show HN上一个2分的项目，0评论。几乎没人注意到。

但如果你在关注"AI写代码你敢直接跑吗"这个问题——Pure Effect可能是今年最重要的发现之一。一个不到1KB（gzipped）的TypeScript effect库，零依赖，6个原语，一个下午就能学会。

它的核心概念简单到让人怀疑：业务逻辑不执行I/O，而是返回一个"我要做什么"的纯数据描述。在真正执行之前，可以被完整审查。

为什么这和AI代码安全有关

2026年Vibe Coding的核心争议：AI生成的代码质量不可预测。你怎么知道它没有在某个分支里删了你的数据库、用了未授权的外部API、把敏感数据写进了日志？

目前的做法是用另一个LLM来审查LLM的代码。但社区正在形成新共识：不该用LLM审查LLM。应该用确定性机制。

AI生成的代码不直接执行I/O → 返回一个effect tree（纯数据）→ 在interpreter运行前walk整个tree → 看到所有副作用，决定是否运行。等于给了AI一把"只读枪"。

六个原语

原语做什么例子
Success(value)返回成功结果Success(user)
Failure(error)停止管道Failure('邮箱已注册')
Command(cmd, next)描述I/O操作Command(findUser, …)
Ask(next)读取上下文Ask((ctx) => …)
Retry(effect, opts)失败重试Retry(cmd, {attempts:3})
Parallel(effects, next)并发执行Parallel([cmd1, cmd2])

注册用户流程示例

const registerUserFlow = (input) =>
  effectPipe(
    validateRegistration,       // 纯函数：查邮箱格式
    () => findUserByEmail(input.email),  // Command：查用户
    ensureEmailAvailable,       // 纯函数：判是否已注册
    () => saveUser(input)      // Command：保存用户
  )(input);

// 测试不是"跑结果对不对"，而是"这个tree对不对"
const flow = registerUserFlow(input);
assert.equal(flow.cmd.name, 'cmdFindUser');
// 你在审查意图，不是审查效果

生产Bug复现不需要数据库

传统async/await：报错→一个堆栈，不知道哪个调用触发了问题。Pure Effect：每个I/O都记录为数据，本地精确重放生产环境的每一步。还有onBeforeCommand hook做运行时guardrail。

和Effect-TS的关键区别

Effect-TS是完整框架，需要按它的方式组织整个代码库。Pure Effect是库——不到1KB，可以渐进式采用，一个函数一个函数地改，不用重写整个项目。

2026下半场共识

ANMA（YAML合同约束）、Pure Effect（Effect系统审查）、State Harness（Lyapunov数学检测）——三种完全不同的方法，指向同一个方向：确定性 > 概率性。

不迷信"更好的模型会自己变安全"。用规则、类型、数学来约束行为。
