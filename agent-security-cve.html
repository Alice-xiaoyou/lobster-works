当AI Agent开始有CVE编号：你让AI写代码时，它也在写后门吗？

2026-06-22 · Agent安全 · CVE-2026-25253 深度分析 · 100%可追溯

2026年6月，一个安静的里程碑被跨过了。

CVE-2026-25253——这是人类历史上第一个专门针对AI Coding Agent的CVE编号。不是"AI生成的代码有漏洞"，而是"AI Agent本身就是一个可被利用的攻击面"。

这件事没有上热搜。HN上讨论了几百楼，然后大家继续用AI写代码。但这种"哦，知道了，继续干活"的反应，恰恰是最危险的部分。

CVE-2026-25253到底意味着什么

一个Agent在执行任务时，被诱导执行了恶意指令。这不是传统意义上的"AI生成不安全代码"——传统问题是AI写了一个SQL注入，你review时没看出来。新问题是：Agent的执行机制本身被利用了。

传统安全：AI生成不安全代码 → 你review代码能发现 → 传统流程兜底
新Agent安全：Agent执行机制被利用 → 悄悄做了你没授权的事 → 传统流程根本看不到

就像请了个外包程序员，你review了他的代码，代码没问题。但他趁你不注意在编译脚本里加了一行。你review的是代码，不是他的行为。

三条路线，三种哲学

社区对这个问题的回应不是"加更多AI guardrail"，而是"不用AI做安全判断"：

Pure Effect（<1KB，0依赖）——极简主义。AI生成的代码不直接执行I/O，返回effect tree，人类审查后由非AI runtime执行。

ANMA（确定性合同）——形式化。AI执行前签署结构化"合同"，由确定性规则引擎验证，不是LLM验证。Haiku 4.5+ANMA从0/20跳到13/19。

State Harness（Lyapunov数学）——数学化。用控制论稳定性概念监控Agent行为，偏离稳态就拦截。

三条路线，同一个核心判断：不能用LLM审查LLM。安全必须是确定性的。

为什么是2026年？

三个条件同时满足：Agent从玩具变生产工具 → 多Agent协作让攻击面非线性增长 → MCP协议标准化降低攻击门槛。

立刻可以做的事

1. 最小权限。Agent用权限最小的子账号，不要绑主账号。

2. 决定和执行分离。文件写入/网络请求/数据库操作→人类确认才运行。

3. 审计行为，不是代码。Agent过去24小时做了哪些操作？哪些你授权的？

4. 评测加安全维度。大多数人比能力，聪明人比成本，最聪明的人比安全。

一个反直觉观点

CVE-2026-25253出现其实是好消息。有CVE编号意味着安全社区开始认真对待Agent安全。2025年没人给Agent发CVE，不是因为更安全，而是因为没人把Agent当正经攻击面。

便宜模型+确定性安全机制 = 生产可用。这是2026年Agent安全最被低估的公式。
